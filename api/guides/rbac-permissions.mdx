## Overview

Every API request includes an `x-role-type` header that determines what the caller can do. Your API token must belong to a user with the specified role in the organization.

<Tip>
**For 3rd party integrations, use `CLIENT_ORG_ADMIN`.** It has full access to all grant, wallet, employee, and organization endpoints.
</Tip>

---

## Roles

| Role | Description | Typical Use |
|------|-------------|-------------|
| `CLIENT_ORG_ADMIN` | Full administrative access | 3rd party integrations, HRIS sync |
| `PAYROLL_ADMIN` | Payroll and settlement operations | Payroll systems |
| `FINANCE_ADMIN` | Financial reporting and payments | Accounting integrations |
| `INVESTOR` | Read-only investment data | Investor portals |

## How Permissions Work

1. **Token scoping**: Each API token is tied to a specific user and organization
2. **Role verification**: The `x-role-type` header is validated against the token's user roles
3. **If the role doesn't match**: You get a `403 Forbidden` error:

```json
{
  "error": "Invalid token or insufficient permissions",
  "details": "User does not have role CLIENT_ORG_ADMIN in organization abc-123"
}
```

## Minimum Required Role

For a complete 3rd party integration that manages grants, wallets, and employees, you need **`CLIENT_ORG_ADMIN`**. This is the only role with write access to all critical endpoints.

If your integration only needs to read data (e.g., a reporting dashboard), `FINANCE_ADMIN` or `PAYROLL_ADMIN` may be sufficient with more limited scope.
